PROJECT OVERVIEW
================================================================================
This project implements a Machine Learning-based predictive maintenance system
using decision trees to identify machine failures early and diagnose their root
causes. The system processes real-time sensor data and provides predictions with
configurable business logic based on operational priorities.

Dataset: Predictive Maintenance Dataset (10,000 samples)
- Training set: 8,000 samples (80%)
- Test set: 2,000 samples (20%)
- Target: Machine failure (binary classification)
- Failure types: 6 classes (0=No failure, 1=TWF, 2=HDF, 3=PWF, 4=OSF, 5=RNF)


I. DECISION TREES VS NAIVE BAYES COMPARISON
================================================================================

1. DECISION TREES (SELECTED)
   ─────────────────────────────────────────────────────────────────────────

   Advantages:
   ✓ Interpretability: Clear decision paths showing feature importance
   ✓ Non-linear relationships: Captures complex interaction patterns
   ✓ No feature scaling required: Works with raw sensor data
   ✓ Mixed data types: Handles both continuous and categorical features
   ✓ Feature importance: Built-in feature importance scoring
   ✓ Threshold tuning: Precise control via probability thresholds
   ✓ Imbalanced data: Works well with SMOTE augmentation
   ✓ Multi-output support: Binary, multilabel, and multiclass capability
   
   Performance on Dataset:
   - Binary DT (13 features) @ 0.96 threshold:
     * Precision: 0.7067 | Recall: 0.7794 | F1: 0.7413
     * CV (5-fold): Precision 0.6942±0.0922, Recall 0.7343±0.0568
   
   - Multilabel DT (8 features) @ 0.96 threshold:
     * Precision: 0.6667 | Recall: 0.6176 | F1: 0.6412
     * CV (5-fold): Precision 0.5808±0.0392, Recall 0.6899±0.0387


2. NAIVE BAYES (NOT SELECTED)
   ─────────────────────────────────────────────────────────────────────────

   Disadvantages for this use case:
   ✗ Conditional independence assumption: Assumes features are independent,
     which is false for sensor data (e.g., temperature correlates with torque)
   ✗ Probabilistic output only: No interpretable decision rules
   ✗ Poor with imbalanced data: Struggles without careful balancing
   ✗ Feature scaling sensitivity: Requires preprocessing and normalization
   ✗ Limited feature interactions: Cannot capture non-linear relationships
   ✗ Binary classification focus: Multiclass is less intuitive
   ✗ Threshold tuning: Less flexible threshold control
   
   Estimated performance on dataset (based on literature):
   - Would achieve ~0.50-0.60 F1 score on this imbalanced dataset
   - Higher false negatives (missed failures) = operationally risky
   - Less explainable: Cannot justify why a failure was predicted

II. WHY DECISION TREES WERE SELECTED
================================================================================

1. DOMAIN REQUIREMENTS
   ─────────────────────────────────────────────────────────────────────────
   
   a) Explainability in Manufacturing:
      - Engineers need to understand WHY a failure is predicted
      - DTs provide clear decision rules showing feature thresholds
      - Naive Bayes provides only probability values with no reasoning
   
   b) Actionability:
      - Maintenance teams must act on predictions
      - DT feature importance guides preventive interventions
      - Example: "Reduce temperature by 5°C" → clear action
   
   c) Risk Management:
      - Missed failures (false negatives) = equipment damage
      - DT's high recall (0.78) minimizes missed failures
      - Configurable thresholds allow safety-first tuning


2. DATA CHARACTERISTICS
   ─────────────────────────────────────────────────────────────────────────
   
   a) Imbalanced Dataset:
      - 97% non-failure samples vs 3% failure samples
      - DTs + SMOTE: Effective combination for this scenario
      - Naive Bayes: Struggles without balanced data
   
   b) Sensor Data Properties:
      - Non-linear relationships (e.g., temperature × speed interactions)
      - DTs capture these automatically; NB assumes independence
      - Derived features (Temp_Diff, Power_Estimate, etc.) boost DT performance
   
   c) Feature Types:
      - Continuous (temperature, speed) + Categorical (machine type)
      - DTs handle both natively
      - NB requires additional preprocessing


3. OPERATIONAL FLEXIBILITY
   ─────────────────────────────────────────────────────────────────────────
   
   a) Threshold Tuning:
      - Different business priorities require different thresholds
      - Safety-first mode: Threshold 0.75 → High recall, tolerate false positives
      - Cost-optimization mode: Threshold 0.96 → High precision, avoid false maintenance
      - NB: Cannot achieve this level of control
   
   b) Multi-Output Capabilities:
      - Binary: Failure or no failure
      - Multilabel: Multiple concurrent failures
      - DTs support all; NB is primarily binary
   
   c) Real-time Deployment:
      - DTs are fast and deterministic
      - No probabilistic distribution assumptions to violate
      - Edge device deployment is straightforward


4. PERFORMANCE METRICS
   ─────────────────────────────────────────────────────────────────────────
   
   Binary DT (13 features) @ Threshold 0.96:
   • Test Precision: 0.7067 (70.67%)
     → When model predicts failure, it's correct 70.67% of the time
   
   • Test Recall: 0.7794 (77.94%)
     → Catches 77.94% of actual failures (only 22% missed)
   
   • F1 Score: 0.7413
     → Balanced performance across both metrics
   
   • Confusion Matrix (Test):
     ├─ True Negatives: 1910 (correctly identified no failure)
     ├─ True Positives: 53 (correctly identified failure)
     ├─ False Positives: 22 (unnecessary maintenance predicted)
     └─ False Negatives: 15 (failures missed - RISK)
   
   • Cross-Validation (5-fold):
     ├─ Mean Precision: 0.6942 ± 0.0922
     ├─ Mean Recall: 0.7343 ± 0.0568
     └─ Mean F1: 0.7062 ± 0.0213

IV. COMPLETE PROJECT WORKFLOW
================================================================================

1. DATA PIPELINE
   ─────────────────────────────────────────────────────────────────────────

   Phase 1: Data Collection & Loading
   ┌─────────────────────────────────┐
   │ Source: Predictive_M.csv        │
   │ Records: 10,000 samples         │
   │ Features: 6 base + 3 one-hot    │
   └──────────────┬──────────────────┘
                  │
   Phase 2: Data Splitting
   ├─ Stratified Split (80/20)
   ├─ Random state = 42 (reproducibility)
   ├─ Train: 8,000 samples
   └─ Test: 2,000 samples

   Phase 3: Feature Engineering (Training Set)
   ├─ Base Features (8):
   │  ├─ Air temperature [K]
   │  ├─ Process temperature [K]
   │  ├─ Rotational speed [rpm]
   │  ├─ Torque [Nm]
   │  ├─ Tool wear [min]
   │  ├─ Type_H (one-hot)
   │  ├─ Type_L (one-hot)
   │  └─ Type_M (one-hot)
   │
   ├─ Engineered Features (5):
   │  ├─ Temp_Diff = Process Temp - Air Temp
   │  ├─ Speed_Torque_Product = RPM × Torque
   │  ├─ Power_Estimate = (RPM/1000) × Torque
   │  ├─ Wear_Speed_Ratio = Tool Wear / (RPM/1000)
   │  └─ Thermal_Load = Temp_Diff × (RPM/1000)
   │
   └─ Total: 13 features (8 base + 5 engineered)

   Phase 4: Data Preprocessing
   ├─ Scaling: StandardScaler fitted on training
   │  └─ Applied to both train and test
   ├─ SMOTE Augmentation: Applied to training only
   │  └─ Creates synthetic minority samples
   │  └─ Balances class distribution
   └─ Result: SMOTE-augmented training set ready

3. INFERENCE PIPELINE (Runtime)
   ─────────────────────────────────────────────────────────────────────────

   Step 1: Input Reception (Gradio Interface)
   └─ User submits sensor readings + configuration
   └─ Data passed to router.py

   Step 2: Configuration Mapping
   └─ (business_priority, diagnostic_detail) → MODEL_MAP entry
   └─ Retrieves model names, thresholds, metrics

   Step 3: Data Preprocessing
   ├─ Feature engineering: calculate derived features
   ├─ One-hot encoding: machine type
   └─ Scaling: using training set statistics (pre-fitted)

   Step 4: Binary Failure Gate
   ├─ Load failure detection model (13 features)
   ├─ Generate probability predictions
   ├─ Apply threshold (0.75 or 0.96)
   ├─ Output: failure_predicted (bool), failure_confidence (float)
   │
   └─ If NO failure predicted:
      └─ Return "No failure" with confidence

   Step 5: Conditional Cause Analysis
   ├─ If failure predicted:
   │  ├─ Use default diagnostic mode: "All contributing causes"
   │  ├─ Load multilabel cause model
   │  └─ Generate cause predictions & confidence
   │
   └─ Only if binary gate passed (prevents over-diagnosis)

   Step 6: Output Formatting
   ├─ Prediction: failure_predicted, failure_reason, confidence
   ├─ Model Info: model_name, threshold, failure_threshold
   └─ Metrics: precision, recall, F1 (from config)

   Step 7: Gradio Display
   ├─ Result card with prediction and confidence
   ├─ Metrics table for transparency
   ├─ Feature breakdown with decision information
   └─ Recommendations based on prediction

V. KEY INSIGHTS & CONCLUSIONS
================================================================================

1. WHY THIS APPROACH WORKS
   ─────────────────────────────────────────────────────────────────────────
   • Sensor data exhibits non-linear failure patterns → DTs excel
   • Class imbalance handled by SMOTE augmentation
   • Feature engineering captures domain knowledge
   • Configurable thresholds provide operational flexibility
   • Transparent decision-making suitable for regulated environment


2. ACHIEVED PERFORMANCE
   ─────────────────────────────────────────────────────────────────────────
   • 77.94% recall: Catches almost 4 out of 5 real failures
   • 70.67% precision: False alarms are acceptable trade-off
   • Balanced F1 of 0.74: Good overall performance
   • Cross-validation std ±0.02: Stable model


3. LIMITATIONS & FUTURE IMPROVEMENTS
   ─────────────────────────────────────────────────────────────────────────
   • Current dataset is historical; real-time data may shift
   • ~23% false negative rate requires further tuning
   • Could benefit from ensemble methods (Random Forest)
   • Deep learning could capture temporal patterns if time-series available
   • Explainability could be enhanced with LIME/SHAP


4. BUSINESS IMPACT
   ─────────────────────────────────────────────────────────────────────────
   • Reduce unplanned downtime by early failure detection
   • Optimize maintenance schedule based on predictions
   • Two modes support different operational strategies
   • Transparent predictions enable operator trust
